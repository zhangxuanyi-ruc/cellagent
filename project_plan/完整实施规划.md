# CellAgent 完整实施规划

## 一、项目总览

CellAgent 是一个以多模态模型先验为起点、以 DE 和知识证据为校验核心、通过 REACT + Reflection 迭代修正的细胞注释与功能解释 Agent。

### 核心流程

```
原始数据 + metadata
    → 预处理 (Leiden聚类 + DE)
    → 多模态LLM初步推断 (celltype, function)
    → Deterministic Reasoning (拆包 → 标准化 → RAG证据包)
    → REACT验证循环 (Judge打分 → 反思)
    → 最多3轮迭代
    → 输出完整结果卡
```

### 统一数据流契约

CellAgent 采用“配置启动 + manifest 传递产物 + AgentState 承载运行状态”的数据流，不在下游阶段硬编码路径，也不在 agent 状态中传递大型 `AnnData/DataFrame` 对象。

1. `config/config.yaml`
   - 只保存用户入口参数和全局默认配置，例如 `pipeline.input_path`、`pipeline.output_root`、GPU、LLM endpoint、RAG 参数。
   - 不作为阶段产物的唯一事实来源。

2. `pipeline_manifest.json`
   - 由 `notebooks/01_sca.ipynb` 在 SCA 阶段结束时写入 `pipeline.output_root/pipeline_manifest.json`。
   - 记录本次 run 的真实产物路径，包括预处理 h5ad、QC report、feature npz、cluster csv、DE summary、多模态先验 JSON、reasoning 输出目录、final 输出目录和 config snapshot。
   - 后续 notebook / script / agent 节点统一读取该 manifest，避免根据 config 重复推断路径。

3. `AgentState`
   - 只保存轻量、可序列化的运行状态：`cluster_id`、metadata、DE csv 路径、top DE genes、多模态预测、reasoning evidence、score、reflection history。
   - 大型矩阵和表格只通过 manifest 中的路径按需读取。

4. 阶段输出目录

```text
output_root/
  pipeline_manifest.json
  preprocessed/
    *_preprocessed.h5ad
  qc_report/
    preprocessing_summary.json
    qc_metrics_summary.csv
  features/
    *_cell_features.npz
  clustering/
    *_cell_clusters.csv
    *_clustering_metrics.json
  de/
    de_summary.csv
    cluster_*_vs_all.csv
  multimodal_prior/
    cluster_predictions.jsonl
  reasoning/
    cluster_*_reasoning.json
    reasoning_summary.json
  final/
    annotations.csv
    annotations.json
```

**设计结论**:
- 路径默认值来自 config，但运行时事实以 manifest 为准。
- agent 内部传递 `AgentState`，不传递已加载的 `AnnData`。
- notebook 可以交互式加载对象调试，但正式阶段接口必须落盘为 JSON/CSV/NPZ/H5AD。

## 三、Phase 1: 标准化层（名称统一映射）

### 目标 ✅ 已完成

所有数据源使用统一的语言：
- **基因名称**: 统一为 HGNC 官方 symbol（人）/ MGI symbol（鼠）— 44986 approved, 126609 aliases
- **细胞类型名称**: 统一为 Cell Ontology ID (`CL:0000000`) — 3434 terms, 6935 映射
- **组织名称**: 统一为 Uberon ID (`UBERON:0000000`) — 14675 terms, 50915 映射
- **功能术语**: 统一为 Gene Ontology ID (`GO:0000000`) — 38560 terms, 138131 映射

### 3.1 `src/tools/rag/mapper.py` — 标准化映射器

**输入**: 各数据库原始的基因名/细胞类型名
**输出**: 标准化后的 HGNC/MGI symbol、CL ID、GO ID

**核心功能**:

1. **基因名称标准化 `normalize_gene(gene_name, species='human') -> str`**
   - 输入: 原始基因名（可能含大小写不一致、旧symbol、别名）
   - 处理: 查询 HGNC (人) / MGI (鼠) 映射表
   - 输出: 官方 approved symbol
   - 依赖: 需预建 `rag/mappers/gene_symbol_mapper.json`（从 HGNC/MGI 官方下载构建）

2. **细胞类型名称标准化 `normalize_cell_type(cell_type_name) -> CL_ID`**
   - 输入: 原始细胞类型文本（如 "T cell", "T cells", "t-cell"）
   - 处理:
     a. 大小写统一、去标点
     b. 查询 CellMarker/PanglaoDB 内置同义词表
     c. 用 CL OBO 的 `name` 和 `synonym` 字段匹配
   - 输出: `CL:0000084`（或 None）
   - 预构建: `rag/mappers/cell_type_to_cl.json`

3. **GO 术语标准化 `normalize_function(function_text) -> List[GO_ID]`
   - 输入: 功能描述文本
   - 处理: 文本匹配 GO term 名称和 synonym
   - 输出: 匹配的 GO ID 列表

4. **跨库细胞类型对齐 `cross_reference_cell_type(name, source) -> CL_ID`**
   - 将 CellMarker 的 "CD4+ T cell"、PanglaoDB 的 "T cells" 映射到同一个 CL ID

### 4.7 `src/tools/rag/__init__.py` — RAGFacade

统一暴露给 engine 的接口：

```python
class RAGFacade:
    def __init__(self, config_path: str, mapper: Mapper)
    
    # Marker 证据
    def query_markers(self, cell_type_cl_id: str, species: str = "human") -> MarkerEvidence
    
    # 功能证据 (GO terms)
    def query_functions(self, cell_type_cl_id: str) -> List[FunctionRecord]
    
    # 组织分布证据
    def query_tissues(self, cell_type_cl_id: str, gene: str = None) -> TissueEvidence
    
    # 细胞类型层次 (用于 fallback 到父类)
    def get_parent_cell_type(self, cl_id: str) -> Optional[str]
    
    # 反查：基因属于哪些细胞类型
    def query_cell_types_for_gene(self, gene: str) -> List[str]
```

**Pydantic Schema**:
```python
class MarkerEvidence(BaseModel):
    cell_type_cl_id: str
    supporting_markers: List[str]           # 正向 marker
    conflicting_markers: List[str]          # 强排他性 marker (指向其他细胞类型)
    sources: List[Literal["cellmarker", "panglaodb"]]

class FunctionRecord(BaseModel):
    go_id: str
    go_name: str
    evidence: str

class TissueEvidence(BaseModel):
    cell_type_cl_id: str
    tissues: List[str]                      # 出现的组织
    source: Literal["hpa", "tabula_sapiens", "immgen"]
```

3. 实现预聚合缓存（parquet）
4. 实现 RAGFacade 统一接口
5. 所有 loader 集成 `mapper.py` 标准化

### 4.7 `src/tools/go_annotation.py` — DE 基因 GO 注释查询 (新增)

**职责**: 从 DE 基因查询关联的 Gene Ontology terms，为功能一致性验证提供基因表达谱层面的证据。

**数据来源**: mygene.info API (免费，无需 key)
- URL: https://mygene.info/v3/query
- 字段: `go.BP` (生物学过程), `go.MF` (分子功能), `go.CC` (细胞组分)

**核心接口**:
```python
class GOAnnotator:
    def query_gene(self, gene_symbol: str) -> dict
        # 查询单个基因的 GO 注释 (GET 请求)

    def query_genes(self, gene_symbols: List[str]) -> Dict[str, dict]
        # 批量查询 — 优先使用 mygene.info POST 批量 API（一次请求查所有基因）
        # fallback 到串行查询

    def summarize_for_judge(self, gene_symbols, aspect="BP", top_k_per_gene=3) -> str
        # 生成供 LLM judge 使用的摘要文本
```

**批量查询优化**:
- mygene.info 支持 `POST /v3/query` with `q=CD3D,CD4,FOXP3&scopes=symbol&fields=go`

## 七、Phase 5: REACT 引擎核心（最关键）

### 7.0 当前实现边界：Reasoning 与 Judge 分层

从当前版本开始，Phase 5 明确拆成两个层：

1. **Deterministic Reasoning 层**
   - 不调用 LLM。
   - 只做规则化处理：多模态先验拆包、名称标准化、DE gene 标准化、按 cell type 检索三类证据、证据结构化落盘。
   - 输出 `ReasoningResult JSON`，作为后续 judge / reflection 的唯一证据输入。

2. **Judge / Reflection 层**
   - 可以使用 LLM。
   - 只消费 ReasoningResult，不重新检索数据库。
   - marker 正向匹配优先规则打分；功能语义、组织合理性、反向 marker 冲突可由 LLM judge 裁决。

这个分层的原因是：Reasoning 输入输出高度模板化，引入 LLM 只会增加幻觉风险；LLM 的价值在于语义裁决和反思修正，而不是路径解析或数据库检索。

### 7.2 `src/core/engine.py` — REACT 状态机

```python
class CellAnnotationEngine:
    def __init__(self, llm: LLMClient, rag: RAGFacade, 
                 config: EngineConfig, mapper: Mapper)
    
    def annotate(self, target_cell_id: str, adata: AnnData, 
                 metadata: dict, initial_llm_result: dict) -> Trace:
        """
        主入口：对单个目标细胞执行完整注释流程
        """
        trace = Trace(trace_id=uuid4(), target_cell_id=target_cell_id, ...)
        
        for round_num in range(1, 4):  # 最多3轮
            # Step 1: Reasoning - 提炼待验证声明
            observation = self._reasoning(initial_llm_result, metadata, de_genes)
            
            # Step 2: Retrieval - 检索所有数据库
            observation = self._retrieve_evidence(observation)
            
            # Step 3: Observation - 整理结构化证据
            observation = self._structure_evidence(observation)
            
            # Step 4: Scoring - 打分
            score = self._score(observation)
            
            # Step 5: Action - LLM 决策
            action = self._llm_decide(observation, score)
            
            trace.rounds.append(RoundRecord(...))
            
            if score.total >= 80 and not score.veto_triggered:
                # 通过验证
                trace.final_output = self._build_final_result(trace)
                trace.status = "SUCCESS"
                return trace
            
            if round_num == 3:
                # 3轮未通过，fallback 到父类
                trace.final_output = self._fallback_to_parent(trace)
                trace.status = "FALLBACK"
                return trace
            
            # 进入下一轮 Reflection
            initial_llm_result = self._reflection(trace, observation, score)
        
        return trace
    
    def _reasoning(self, ...) -> Observation:
        """提炼关键声明，明确需要检索的证据"""
    
    def _retrieve_evidence(self, obs: Observation) -> Observation:
        """并行检索所有数据库 + DE 基因 GO 证据"""
        # 并行调用:
        # rag.query_markers(cell_type_cl_id)
        # rag.query_functions(cell_type_cl_id)      # 辅助参考
        # rag.query_tissues(cell_type_cl_id)
        # rag.query_cell_types_for_gene(gene) for each DE gene
        # go_annotator.summarize_for_judge(de_genes)  # DE 基因 GO 证据（核心）
    
    def _structure_evidence(self, obs: Observation) -> Observation:
        """将原始检索结果整理为结构化证据"""
    
    def _score(self, obs: Observation) -> ScoreCard:
        """
        按权重打分:

        【关键设计变更】功能一致性不再比对"假设类型"的 GO terms,
        而是比对 "DE 基因表达谱" 暗示的功能:
          1. 通过 GOAnnotator 查询每个 DE 基因关联的 GO terms
          2. 汇总得到 "基因表达谱暗示的功能列表"
          3. 将 LLM 功能描述 vs 基因表达谱功能 喂给 LLM judge 打分
          4. 这样即使类型假设错了, 功能打分也不会虚高

        - marker_match: 40分 (规则判断)
        - conflict: 30分 (LLM裁决反向marker)
        - function: 20分 (LLM judge: LLM描述 vs DE基因GO证据)
        - tissue: 10分 (LLM判断组织分布)
        - 一票否决条件: 强排他性冲突 / 功能完全错误 / marker全未匹配
        """
    
    def _llm_decide(self, obs: Observation, score: ScoreCard) -> Action:
        """LLM 根据证据和分数做决策"""
    
    def _reflection(self, trace: Trace, obs: Observation, 
                    score: ScoreCard) -> dict:
        """
        定位失败原因，输出多个可能正确类型
        焦点 (Focus) 机制：下一轮只围绕这些可能类型验证
        """
    
    def _fallback_to_parent(self, trace: Trace) -> FinalResult:
        """3轮未通过，查找CL父类作为fallback输出"""
        # 用 CL 层次关系，从当前类型向上找最近的父类
```

## 十二、关键设计决策

### 决策 1: 功能一致性验证基于 DE 基因 GO 证据，而非假设类型 GO

**问题**: 如果细胞类型假设错误（如把 Treg 误判为 Naive T cell），查假设类型的 GO terms 会得到匹配的 GO terms，导致功能打分虚高（错上加错）。

**方案**:
- 通过 `GOAnnotator` (mygene.info API) 查询每个 DE 基因关联的 GO terms
- 汇总 DE 基因的 GO terms → 得到 "基因表达谱暗示的功能"
- 功能一致性打分: LLM judge 比对 "LLM 功能描述" vs "DE 基因 GO 证据"
- 这样即使类型假设错了，功能打分仍能检测出矛盾

**示例**:
```
假设类型: Naive T cell
LLM 功能描述: "参与淋巴细胞归巢，表达 CCR7"

旧方案 (查假设类型 GO):
  Naive T cell GO → [lymphocyte migration, CCR7 signaling]
  比对 → 匹配 ✅ → 20分 (虚高!)

新方案 (查 DE 基因 GO):
  DE 基因: CD3D, CD4, FOXP3, IL2RA
  FOXP3 GO → [regulatory T cell differentiation]
  IL2RA GO → [regulation of T cell tolerance induction]
  比对 → FOXP3 和 IL2RA 强烈指向 Treg，与 "Naive T" 矛盾 ❌ → 0分
```

### 决策 2: LLM 自由文本输出 + LLM as Judge，而非 Prompt 限制 GO 风格

**问题**: 是否可以通过 prompt 限制多模态 LLM 输出 GO 风格的功能描述？

**对比**:

| 维度 | Prompt 限制 GO 风格 | LLM 自由文本 + Judge |
|---|---|---|
| 可靠性 | 差（LLM 可能 hallucinate GO ID） | 好（只需判断语义） |
| 灵活性 | 差（受 GO 词汇表限制） | 好 |
| 解决问题 1 | 否 | 是（Judge 比对 DE 基因 GO） |

**结论**: LLM 自由文本输出 + LLM as Judge 更可靠。

### 决策 3: `normalize_function()` 的定位

`normalize_function()` 基于文本匹配 GO term 名称，对 LLM 自然语言输出无效（匹配率 0%）。

**保留原因**:
- 研究人员输入关键词时快速查 GO ID
- 构建知识图谱时的索引工具
- 辅助 prompt 构建（提供 GO term 候选）

**明确不用于**: REACT 验证阶段的功能一致性比对。

### 决策 4: CellMarker 复合基因拆分

CellMarker 的 `geneSymbol` 列含复合基因（如 `[CD3D, CD3E, CD3G]`），之前的 loader 将其当作一个字符串处理，导致：
- 基因标准化失败
- 反向查询永远匹配不到

**方案**: 加载时拆分为独立行。原始 ~4万行 → 拆分后 143,819 行，0 残留复合基因。

## 十三、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| CellMarker 备用服务器不稳定 | 无法获取最新marker数据 | 已下载本地备份，定期脚本更新 |
| LLM API 抖动/限流 | REACT 中断 | LLMClient 实现重试 + Mock 模式 |
| CL 名称映射覆盖不全 | 细胞类型无法标准化 | 建立未匹配名称日志，人工补录 |
| 3轮迭代不收敛 | 输出 fallback 父类 | engine 内置 fallback 机制 |
| LLM 幻觉 | 打分/决策不可靠 | 规则打分为主(70%)，LLM打分为辅(30%) |
