# CellAgent 完整实施规划

## 一、项目总览

CellAgent 是一个以多模态模型先验为起点、以 DE 和知识证据为校验核心、通过 REACT + Reflection 迭代修正的细胞注释与功能解释 Agent。

### 核心流程

```
原始数据 + metadata
    → 预处理 (Leiden聚类 + DE)
    → 多模态LLM初步推断 (celltype, function)
    → Deterministic Reasoning (拆包 → 标准化 → RAG证据包)
    → REACT验证循环 (Judge打分 → 反思)
    → 最多3轮迭代
    → 输出完整结果卡
```

### 统一数据流契约

CellAgent 采用“配置启动 + manifest 传递产物 + AgentState 承载运行状态”的数据流，不在下游阶段硬编码路径，也不在 agent 状态中传递大型 `AnnData/DataFrame` 对象。

1. `config/config.yaml`
   - 只保存用户入口参数和全局默认配置，例如 `pipeline.input_path`、`pipeline.output_root`、GPU、LLM endpoint、RAG 参数。
   - 不作为阶段产物的唯一事实来源。

2. `pipeline_manifest.json`
   - 由 `notebooks/01_sca.ipynb` 在 SCA 阶段结束时写入 `pipeline.output_root/pipeline_manifest.json`。
   - 记录本次 run 的真实产物路径，包括预处理 h5ad、QC report、feature npz、cluster csv、DE summary、多模态先验 JSON、reasoning 输出目录、final 输出目录和 config snapshot。
   - 后续 notebook / script / agent 节点统一读取该 manifest，避免根据 config 重复推断路径。

3. `AgentState`
   - 只保存轻量、可序列化的运行状态：`cluster_id`、metadata、DE csv 路径、top DE genes、多模态预测、reasoning evidence、score、reflection history。
   - 大型矩阵和表格只通过 manifest 中的路径按需读取。

4. 阶段输出目录

```text
output_root/
  pipeline_manifest.json
  preprocessed/
    *_preprocessed.h5ad
  qc_report/
    preprocessing_summary.json
    qc_metrics_summary.csv
  features/
    *_cell_features.npz
  clustering/
    *_cell_clusters.csv
    *_clustering_metrics.json
  de/
    de_summary.csv
    cluster_*_vs_all.csv
  multimodal_prior/
    cluster_predictions.jsonl
  reasoning/
    cluster_*_reasoning.json
    reasoning_summary.json
  final/
    annotations.csv
    annotations.json
```

**设计结论**:
- 路径默认值来自 config，但运行时事实以 manifest 为准。
- agent 内部传递 `AgentState`，不传递已加载的 `AnnData`。
- notebook 可以交互式加载对象调试，但正式阶段接口必须落盘为 JSON/CSV/NPZ/H5AD。
- 多模态 prior 是单细胞级输出，因此 agent 必须通过 `cell_id -> clustering cell_clusters.csv -> cluster_id -> de_summary.csv -> cluster DE csv` 找到该细胞所属簇的 DE 证据。

## 三、Phase 1: 标准化层（名称统一映射）

### 目标 ✅ 已完成

所有数据源使用统一的语言：
- **基因名称**: 统一为 HGNC 官方 symbol（人）/ MGI symbol（鼠）— 44986 approved, 126609 aliases
- **细胞类型名称**: 统一为 Cell Ontology ID (`CL:0000000`) — 3434 terms, 6935 映射
- **组织名称**: 统一为 Uberon ID (`UBERON:0000000`) — 14675 terms, 50915 映射
- **功能术语**: 统一为 Gene Ontology ID (`GO:0000000`) — 38560 terms, 138131 映射

### 3.1 `src/tools/rag/mapper.py` — 标准化映射器

**输入**: 各数据库原始的基因名/细胞类型名
**输出**: 标准化后的 HGNC/MGI symbol、CL ID、GO ID

**核心功能**:

1. **基因名称标准化 `normalize_gene(gene_name, species='human') -> str`**
   - 输入: 原始基因名（可能含大小写不一致、旧symbol、别名）
   - 处理: 查询 HGNC (人) / MGI (鼠) 映射表
   - 输出: 官方 approved symbol
   - 依赖: 需预建 `rag/mappers/gene_symbol_mapper.json`（从 HGNC/MGI 官方下载构建）

2. **细胞类型名称标准化 `normalize_cell_type(cell_type_name) -> CL_ID`**
   - 输入: 原始细胞类型文本（如 "T cell", "T cells", "t-cell"）
   - 处理:
     a. 大小写统一、去标点
     b. 查询 CellMarker/PanglaoDB 内置同义词表
     c. 用 CL OBO 的 `name` 和 `synonym` 字段匹配
   - 输出: `CL:0000084`（或 None）
   - 预构建: `rag/mappers/cell_type_to_cl.json`

3. **GO 术语标准化 `normalize_function(function_text) -> List[GO_ID]`
   - 输入: 功能描述文本
   - 处理: 文本匹配 GO term 名称和 synonym
   - 输出: 匹配的 GO ID 列表

4. **跨库细胞类型对齐 `cross_reference_cell_type(name, source) -> CL_ID`**
   - 将 CellMarker 的 "CD4+ T cell"、PanglaoDB 的 "T cells" 映射到同一个 CL ID

### 3.5 Marker 证据库统一映射与缺失处理（最新决策）

此前为解决少数胰腺细胞类型查不到 marker，曾尝试在推理期加入 tissue-context alias 和 source-specific alias。复盘后决定撤销这一路线：第一版系统不应对个别细胞类型写硬编码补丁，也不应在推理期动态猜测数据库别名。Marker 证据应在构建期统一标准化，推理期只做确定性查询。

#### 设计原则

1. **Cell Ontology 仍是唯一细胞类型标准锚点**
   - 多模态 prior 的 `cell_type` 先标准化为 CL ID。
   - Marker、function、tissue 三类证据都围绕 CL ID 组织。
   - 不因为 CellMarker/PanglaoDB 命名不一致而绕开 CL。

2. **Marker 库职责保持单一**
   - Marker 主库：CellMarker 2.0。
   - Marker fallback：PanglaoDB。
   - HPA 和 Tabula Sapiens 不进入 marker 打分；Tabula Sapiens 只做组织分布主库。

3. **构建期解决数据库名称对齐**
   - 读取 CellMarker 和 PanglaoDB 原始 cell type 名称。
   - 用 `cell_type_to_cl.json` 标准化到 CL ID。
   - 能标准化到 CL 的 marker 统一聚合进 `CL ID -> marker genes` mapper。
   - 标准化失败的数据库名称进入 audit 报告，人工决定是否补充 mapper，而不是在推理期临时猜 alias。

4. **推理期不做特化 alias**
   - 不再保留 `pancreatic acinar cell -> acinar cell` 这类硬编码规则。
   - 不再保留 `pancreatic ductal cell -> ductal cells` 这类 source-specific runtime alias。
   - 不做全局删除组织/器官词，因为会把精确标签压成歧义短名。

5. **marker 缺失不是 prior 错误**
   - 如果 CellMarker 和 PanglaoDB 统一 mapper 都没有该 CL 的 marker，则标记为 `INSUFFICIENT_DATABASE_EVIDENCE`。
   - 该状态不能被 LLM 用生物学常识补齐 marker 分。
   - 该状态触发 reflection candidate generation。

当出现以下情况时进入 reflection：

```text
marker_status == INSUFFICIENT_DATABASE_EVIDENCE
或 marker_score == 0
```

Reflection 不是让 LLM 直接给最终答案，而是让 agent LLM 生成候选包：

```text
输入:
  original multimodal prior cell_type/function
  cluster_id
  target cell_id
  cluster top10 DE genes
  cluster top30 DE genes
  dataset name / disease / tissue metadata
  marker failure reason

输出:
  candidate cell types
  candidate functions
  candidate tissue/context
  evidence rationale
```

输出 JSON：

```json
{
  "trigger": "INSUFFICIENT_DATABASE_EVIDENCE",
  "original_prior": {
    "cell_type": "...",
    "function": "..."
  },
  "cluster_evidence": {
    "cluster_id": "...",
    "top10_de_genes": ["..."],
    "top30_de_genes": ["..."]
  },
  "candidates": [
    {
      "cell_type": "...",
      "function": "...",
      "tissue_context": "...",
      "reason": "...",
      "confidence": 0.0
    }
  ]
}
```

每个 reflection candidate 必须重新进入同一套确定性验证：

```text
candidate.cell_type -> CL ID
CL ID -> marker mapper
query function/tissue evidence
marker/function/tissue/conflict scoring
```

LLM 不允许凭常识直接覆盖最终注释；它只负责提出候选，最终是否接受仍由证据评分决定。

#### 第一轮测试空间

第一轮 marker scoring 和 reflection 测试优先使用训练标签空间中常见且 marker 可覆盖的类型：

```text
T cell
B cell
monocyte
macrophage
natural killer cell
endothelial cell
fibroblast
epithelial cell
pancreatic A cell
type B pancreatic cell
pancreatic acinar cell
pancreatic ductal cell
```

暂不以任意完整 CL term 或明显训练空间外的超细粒度 label 作为核心测试，例如：

```text
activated CD8-positive alpha-beta T cell
```

这类标签可以作为后续 robustness 测试，但不能混入第一轮主流程验证。

#### Marker scoring v2 决策

Marker 维度拆成正向 marker 支持和反向 marker 冲突两部分。正向 marker 使用确定性规则打分；反向 marker 必须使用 LLM-as-judge 才能触发 veto。若 LLM judge 未启用，系统只记录反向候选证据，不做规则 veto，避免共享 marker、混合簇、ambient RNA 或 doublet 导致误杀。

正向 marker 规则层必须先计算：

```text
top10_match_count
top30_match_count
top10_matches
top30_matches
top10_misses
top30_misses
marker_gene_list
marker_evidence_source
```

正向 marker 打分：

```text
top10_match >= 5
  -> 40

top10_match 3-4
  -> 30

top10_match 1-2
  -> 10

top10_match == 0
  -> 0

top30_rescue:
  top30_match >= 10 -> 30
  top30_match 6-9  -> 20
  top30_match 3-5  -> 10
  top30_match 0-2  -> 0

marker_score = max(top10_score, top30_rescue_score)
```

正向 marker 解释：

```text
>=5 top10 hits: strong support
3-4 top10 hits: moderate support
1-2 top10 hits: weak support，不单独通过
0 hits: no marker support；若 marker 数据库充分，进入后续 veto/反思判断
marker list 为空: INSUFFICIENT_DATABASE_EVIDENCE，不能等价为预测错误
```

反向 marker 候选生成：

```text
primary_reverse_candidates =
  (top10 DE genes - predicted cell type marker hits)
  -> query_cell_types_for_gene(gene)
  -> gene + candidate_cl_id + candidate_cell_type_name + evidence_scope="top10_primary"

exclusive_veto_screen_candidates =
  (top30 DE genes - predicted cell type marker hits)
  -> query_cell_types_for_gene(gene)
  -> gene + candidate_cl_id + candidate_cell_type_name + evidence_scope="top30_exclusive_veto_screen"
```

反向 marker 的设计原则：

```text
top10 primary candidates 是主判定证据。
top30 screen candidates 只作为强排他 veto 的辅助排查，不参与普通扣分。
一个普通 marker 不足以重新注释细胞，也不足以直接否决 prior。
反向 marker 只用于高置信错误拦截，不作为开放式细胞类型推断模块。
```

反向 marker LLM judge 规则：

```text
NO_CONFLICT:
  score = 30
  veto = false

WEAK_NOISE:
  score = 15
  veto = false
  用于候选分散、共享 marker、状态 marker、同谱系兼容、混合簇、ambient RNA、doublet 或聚类不纯等情况

STRONG_CONFLICT:
  score = 0
  veto = true
  仅用于明确跨大谱系或同谱系互斥的核心 marker 冲突
```

LLM judge 约束：

```text
只能使用输入中的 genes 和 reverse-marker candidate cell types。
不能引入新的 marker gene、新的 marker 关系或新的 candidate cell type。
可以使用基础细胞生物学知识判断“已提供的 candidate cell type”与 prior 是否兼容或互斥。
如果证据不足，必须输出 WEAK_NOISE 或 NO_CONFLICT，不能 veto。
单基因 veto 只允许用于高特异核心 marker 且 candidate cell type 与 predicted cell type 明确互斥的情况。
两个或更多 top10 unmatched genes 一致指向同一个不兼容 cell type，可以触发 veto。
```

实现位置：

```text
src/agent/tools/judging.py
  _score_markers()
  _score_conflicts()
  _reverse_marker_candidates()
  _exclusive_veto_screen_candidates()

src/llm/prompts.py
  conflict_arbitration_prompt()
```

### 4.7 `src/tools/go_annotation.py` — DE 基因 GO 注释查询 (新增)

## 七、Phase 5: REACT 引擎核心（最关键）

### 7.0 当前实现边界：Reasoning 与 Judge 分层

从当前版本开始，Phase 5 明确拆成两个层：

1. **Deterministic Reasoning 层**
   - 不调用 LLM。
   - 只做规则化处理：多模态先验拆包、名称标准化、DE gene 标准化、按 cell type 检索三类证据、证据结构化落盘。
   - 输出 `ReasoningResult JSON`，作为后续 judge / reflection 的唯一证据输入。

2. **Judge / Reflection 层**
   - 可以使用 LLM。
   - 只消费 ReasoningResult，不重新检索数据库。
   - marker 正向匹配优先规则打分；功能语义、组织合理性、反向 marker 冲突可由 LLM judge 裁决。

这个分层的原因是：Reasoning 输入输出高度模板化，引入 LLM 只会增加幻觉风险；LLM 的价值在于语义裁决和反思修正，而不是路径解析或数据库检索。

**评分层结构约束（TODO 拆分）**:

Reasoning 是四类评分的上级证据构建接口，不属于某一个评分维度。后续评分层应拆成平行 scorer，避免 `judging.py` 变成大杂烩：

```text
ReasoningNode
  -> ReasoningResult 统一证据包

Scoring / Judge Layer
  -> marker_scorer.py    # 正向 marker overlap，规则为主
  -> conflict_scorer.py  # 反向 marker conflict，LLM judge 裁决
  -> function_scorer.py  # function 一致性，LLM judge 裁决
  -> tissue_scorer.py    # tissue 合理性，规则 + LLM judge

JudgeAggregator
  -> 合并四路分数、veto、warnings、final decision
```

当前初版仍由 `src/agent/tools/judging.py` 单文件承载四路评分，但代码和输出必须按四个评分维度组织，后续再拆文件。

**处理步骤**:

```text
Step 2: 标准化
```

### 7.0.4 `src/agent/tools/judging.py` — Judge / Scoring 分支

**输入**:
- `ReasoningResult`
- config 中的 `engine.scoring` 和 `engine.veto`
- 可选 LLM client（当前默认关闭，后续接入 Qwen-27B judge）

**当前实现边界**:
- marker 正向匹配：规则打分，严格使用目标 cell 所属 cluster 的 top10 DE genes 与 RAG 全量 marker list 的 normalized overlap。
- 反向 marker 冲突：从未匹配的 top10 DE genes 反查 marker 指向，只构建 conflict candidates；是否强排他、是否 veto 由 LLM judge 裁决。LLM judge 未启用时，不允许仅凭候选数量触发反向 marker veto。
- function 一致性：保留 LLM judge prompt；未启用 LLM 时，若存在权威功能记录则给中性占位分，不伪造语义判断。
- tissue 一致性：若 metadata tissue UBERON ID 与 RAG tissue ID 直接匹配则满分；否则未启用 LLM 时给中性占位分。
- `run_agent.py --enable-llm-judge` 可开启配置中的 LLM client；默认关闭，保证离线 deterministic smoke test 可运行。

**输出**:

```text
output_root/judging/cluster_{cluster_id}_judge.json
```

**Marker 正向匹配打分规则（40 分）**:

```text
输入:
  top10 DE genes from cluster_{id}_vs_all.csv
  top30 DE genes from cluster_{id}_vs_all.csv
  如果该 cluster 显著 DE gene 数量不足 10，则使用该 csv 中全部已保存显著 DE genes
  marker list from rag.query_markers(predicted_cell_type)

top10_score:
  matched >= 5  -> STRONG_SUPPORT, 40
  matched 3-4   -> MODERATE_SUPPORT, 30
  matched 1-2   -> WEAK_SUPPORT, 10；弱支持不代表单独通过，最终由总分阈值和其他维度决定
  matched 0     -> 0

top30_rescue_score:
  matched >= 10 -> 30
  matched 6-9   -> 20
  matched 3-5   -> 10
  matched 0-2   -> 0

最终:
  marker_score = max(top10_score, top30_rescue_score)
  如果 top30_rescue_score > top10_score，则 marker_status = RESCUED_SUPPORT
  如果 marker_score = 0 且 marker 数据库充分，则 marker_status = NO_SUPPORT，触发 marker veto / reflection
  marker list 为空 -> INSUFFICIENT_DATABASE_EVIDENCE；不能等价为预测错误
```

该规则不要求所有 top10 DE genes 都必须是 marker，因为 cluster DE 会包含状态、应激、代谢、细胞周期等非 cell-type marker；使用“top10 中至少 5 个命中”作为满分阈值更符合常规生信注释实践。top30 rescue 是为测序噪声、DE 排序误差和复杂生物系统保留的宽容分支，但最高只给 30 分，不能替代 top10 强支持满分。

DE 保存阶段已经执行同类 fallback：`de_analysis.top_k=30` 表示“最多保存 30 个显著 DE genes”，如果某个 cluster 经 `adjusted_p_value` 和 `logFC` 过滤后显著基因不足 30，则只保存全部显著基因。Agent marker scoring 再从已保存结果中最多取 top10；若不足 10，继续使用全部可用显著基因，评分阈值不变。这种情况本身提示该 cluster 的差异信号较弱或聚类质量可能不足，因此不额外放宽得分规则。

**TODO: LLM-as-judge for ambiguous marker evidence**:
- 初版 marker 正向匹配仍以规则为主，不让 LLM 接管常规 scoring。
- 后续仅在边界情况调用 LLM-as-judge：
  - top10 hits 为 1-2，属于 `WEAK_SUPPORT`；
  - top10 hits 为 0 但 top30 hits 存在；
  - top10 与 top30 或正反向 marker 证据冲突。
- LLM 输入必须是固定证据包：predicted cell type、cluster top10 DE、cluster top30 DE、full marker list、top10 hits、top30 hits、marker source。
- LLM 输出必须是固定 JSON：`decision, score_recommendation, supporting_genes, conflicting_genes, reason`。
- LLM 不允许引入新 marker、不允许改写 cell type，只能解释给定证据。

**下一步 LLM judge 实现顺序**:
1. `conflict_scorer`：优先实现。输入 predicted cell type、top10/top30 DE、正向 marker hits、reverse marker candidates；输出 `score/veto/veto_reason/reasoning/conflicting_genes`。
2. `function_scorer`：输入多模态 prior 的 function 文本和 Cell Ontology function/definition records；输出 0/10/20 分和是否语义冲突。
3. `tissue_scorer`：输入 metadata tissue/species/context 和 Tabula/ImmGen tissue records；先规则匹配 UBERON ID，复杂情况再调用 LLM judge。

LLM judge 的统一约束：
- 只消费 `ReasoningResult` 中已经检索好的证据，不重新检索数据库。
- 必须 JSON 输出。
- 不允许引入新基因、新 marker、新 cell type。
- LLM 调用失败时使用 deterministic fallback，并在 warnings/provenance 中记录。

### 7.2 `src/core/engine.py` — REACT 状态机

```python
class CellAnnotationEngine:
    def __init__(self, llm: LLMClient, rag: RAGFacade, 
                 config: EngineConfig, mapper: Mapper)
    
    def annotate(self, target_cell_id: str, adata: AnnData, 
                 metadata: dict, initial_llm_result: dict) -> Trace:
        """
        主入口：对单个目标细胞执行完整注释流程
        """
        trace = Trace(trace_id=uuid4(), target_cell_id=target_cell_id, ...)
        
        for round_num in range(1, 4):  # 最多3轮
            # Step 1: Reasoning - 提炼待验证声明
            observation = self._reasoning(initial_llm_result, metadata, de_genes)
            
            # Step 2: Retrieval - 检索所有数据库
            observation = self._retrieve_evidence(observation)
            
            # Step 3: Observation - 整理结构化证据
            observation = self._structure_evidence(observation)
            
            # Step 4: Scoring - 打分
            score = self._score(observation)
            
            # Step 5: Action - LLM 决策
            action = self._llm_decide(observation, score)
            
            trace.rounds.append(RoundRecord(...))
            
            if score.total >= 80 and not score.veto_triggered:
                # 通过验证
                trace.final_output = self._build_final_result(trace)
                trace.status = "SUCCESS"
                return trace
            
            if round_num == 3:
                # 3轮未通过，fallback 到父类
                trace.final_output = self._fallback_to_parent(trace)
                trace.status = "FALLBACK"
                return trace
            
            # 进入下一轮 Reflection
            initial_llm_result = self._reflection(trace, observation, score)
        
        return trace
    
    def _reasoning(self, ...) -> Observation:
        """提炼关键声明，明确需要检索的证据"""
    
    def _retrieve_evidence(self, obs: Observation) -> Observation:
        """并行检索所有数据库 + DE 基因 GO 证据"""

        # rag.query_functions(cell_type_cl_id)      # 辅助参考

        """将原始检索结果整理为结构化证据"""

        而是比对 "DE 基因表达谱" 暗示的功能:
          1. 通过 GOAnnotator 查询每个 DE 基因关联的 GO terms
          2. 汇总得到 "基因表达谱暗示的功能列表"
          3. 将 LLM 功能描述 vs 基因表达谱功能 喂给 LLM judge 打分
          4. 这样即使类型假设错了, 功能打分也不会虚高

        - marker_match: 40分 (规则判断)
        - conflict: 30分 (LLM裁决反向marker)
        - function: 20分 (LLM judge: LLM描述 vs DE基因GO证据)
        - tissue: 10分 (LLM判断组织分布)

        """

        """
```

## 十二、关键设计决策

### 决策 1: 功能一致性验证基于 DE 基因 GO 证据，而非假设类型 GO

**问题**: 如果细胞类型假设错误（如把 Treg 误判为 Naive T cell），查假设类型的 GO terms 会得到匹配的 GO terms，导致功能打分虚高（错上加错）。

**方案**:
- 通过 `GOAnnotator` (mygene.info API) 查询每个 DE 基因关联的 GO terms
- 汇总 DE 基因的 GO terms → 得到 "基因表达谱暗示的功能"
- 功能一致性打分: LLM judge 比对 "LLM 功能描述" vs "DE 基因 GO 证据"
- 这样即使类型假设错了，功能打分仍能检测出矛盾

**示例**:
```
假设类型: Naive T cell
LLM 功能描述: "参与淋巴细胞归巢，表达 CCR7"

旧方案 (查假设类型 GO):
  Naive T cell GO → [lymphocyte migration, CCR7 signaling]
  比对 → 匹配 ✅ → 20分 (虚高!)

新方案 (查 DE 基因 GO):
  DE 基因: CD3D, CD4, FOXP3, IL2RA
  FOXP3 GO → [regulatory T cell differentiation]
  IL2RA GO → [regulation of T cell tolerance induction]
  比对 → FOXP3 和 IL2RA 强烈指向 Treg，与 "Naive T" 矛盾 ❌ → 0分
```

### 决策 2: LLM 自由文本输出 + LLM as Judge，而非 Prompt 限制 GO 风格

**问题**: 是否可以通过 prompt 限制多模态 LLM 输出 GO 风格的功能描述？

**对比**:

| 维度 | Prompt 限制 GO 风格 | LLM 自由文本 + Judge |
|---|---|---|
| 可靠性 | 差（LLM 可能 hallucinate GO ID） | 好（只需判断语义） |
| 灵活性 | 差（受 GO 词汇表限制） | 好 |
| 解决问题 1 | 否 | 是（Judge 比对 DE 基因 GO） |

**结论**: LLM 自由文本输出 + LLM as Judge 更可靠。

### 决策 3: `normalize_function()` 的定位

`normalize_function()` 基于文本匹配 GO term 名称，对 LLM 自然语言输出无效（匹配率 0%）。

**保留原因**:
- 研究人员输入关键词时快速查 GO ID
- 构建知识图谱时的索引工具
- 辅助 prompt 构建（提供 GO term 候选）

**明确不用于**: REACT 验证阶段的功能一致性比对。

### 决策 4: CellMarker 复合基因拆分

CellMarker 的 `geneSymbol` 列含复合基因（如 `[CD3D, CD3E, CD3G]`），之前的 loader 将其当作一个字符串处理，导致：
- 基因标准化失败
- 反向查询永远匹配不到

**方案**: 加载时拆分为独立行。原始 ~4万行 → 拆分后 143,819 行，0 残留复合基因。

## 十三、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| CellMarker 备用服务器不稳定 | 无法获取最新marker数据 | 已下载本地备份，定期脚本更新 |
| LLM API 抖动/限流 | REACT 中断 | LLMClient 实现重试 + Mock 模式 |
| CL 名称映射覆盖不全 | 细胞类型无法标准化 | 建立未匹配名称日志，人工补录 |
| 3轮迭代不收敛 | 输出 fallback 父类 | engine 内置 fallback 机制 |
| LLM 幻觉 | 打分/决策不可靠 | 规则打分为主(70%)，LLM打分为辅(30%) |
